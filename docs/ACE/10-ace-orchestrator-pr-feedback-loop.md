# ACE — PR-Stage Feedback Loop: Harvesting Engineer Review Conversations into Context Items

## Why PR feedback is a special signal source

Topics 7–9 treated PR comments as one of several signal types flowing into the learning pipeline. This topic zooms in on PR feedback specifically, because it has properties that make it worth treating as a distinct pipeline component.

The distinction is this: every other signal in the system is produced *by the orchestrator*. The plan is generated by Goose. The execution summary is written by Goose. Even clarification history is the agent's uncertainty surfaced for human response. PR review comments are produced *entirely by engineers*, with no agent involvement — unmediated human evaluation of the output. That makes them the highest-confidence signal in the system, and the one most worth extracting well.

---

## What's already wired vs what's missing

**Already wired (from Topic 7):**

The `await_pr_approval` node captures `pr_comments` when the decision is `commented` and persists them to `workflows.pr_comments` via `update_pr_comments()`. When the graph loops back into `generate_code`, the comments are also passed directly to the Goose generate-code recipe — this is a live in-session feedback loop, not a learning loop.

**What's missing:**

The current `pr_comments` field is free text accumulated across review rounds with a separator delimiter. It flows into the generate-code recipe for the immediate re-run but never enters the learning pipeline as structured signal. The Reflector receives it as part of a trace but the comments aren't classified before that point.

Two improvements are needed:
1. Structuring `pr_comments` as a JSON array (planned in Topic 7).
2. A classification step that turns raw review text into typed feedback items before the Reflector sees them.

---

## Why classify before the Reflector, not inside it

The Reflector receives the entire workflow trace: plan JSON, execution summary, clarification history, PR comments, and outcome. That is a large, heterogeneous context. If it also has to classify feedback types while simultaneously identifying generalisable patterns, you're asking it to do two cognitively distinct jobs in one pass — and the classification task can anchor its attention in ways that distort pattern extraction.

A separate classification step keeps each component's responsibility clean. It produces a structured input that lets the Reflector focus entirely on "what is the generalisable pattern here?" rather than "what kind of comment is this and what does it mean?"

---

## The structure of a PR review comment as a learning signal

Four feedback types that are meaningfully different for the Curator:

| Type | Example | Learning implication |
|---|---|---|
| **Correction** | "This should use a migration file, not ALTER TABLE inline" | High-value pattern item; directly actionable |
| **Omission** | "You forgot to handle the rollback case" | Gap in the context store or plan; new item needed |
| **Style/convention** | "We use `snake_case` for all migration names" | Codebase convention; `codebase_wide` scope |
| **Acknowledgment** | "LGTM, nice work" | No learning signal; skip |

The classifier is a lightweight LLM call or rule-based tagger. It runs as the first sub-step of `run_learning_pipeline`, before the Evaluator, operating on the structured `pr_comments` JSON array.

**On text format vs JSON format:** The classifier can operate on the separator-delimited raw text format, but with degraded accuracy: round boundaries may be misidentified if comments contain the separator pattern, per-round metadata (actor, timestamp) isn't machine-readable without parsing the separator header, and the classifier must do structural parsing simultaneously with semantic classification. JSON format makes all of that trivial. The raw format is workable as a bootstrap, but adds fragility.

---

## Each review round is an independent learning event

A workflow can have:
- Round 1: initial PR, reviewer comments on missing test coverage
- Round 2: updated PR, reviewer comments on implementation approach
- Round 3: approved

Each round is a discrete feedback event. The general pipeline in Topic 9 processes the trace after terminal state. But waiting for terminal state means you see the final approved outcome — not the specific deficit that caused round 1 to fail. Each re-run cycle carries its own signal: the context that was injected, what the agent did with it, and why the human found it insufficient.

If the store has a gap that caused a failure in round 1, and the engineer corrects it, that gap-and-correction is a learning event regardless of whether the workflow ultimately succeeds. Processing per-round, not just per-completion, preserves this structure.

A round-1 comment addressed in round-2 with subsequent approval is particularly high value: you know what the problem was *and* what the fix looked like. The Reflector can produce a correction-and-resolution pattern from this that it couldn't produce from the final approved state alone.

This is exactly why the structured `pr_comments` JSON array needs per-round entries — each element has `round`, `comments`, `actor`, and `timestamp` — so the per-round processing can iterate the array rather than parse delimited text.

## The rejection case and `audit_log`

PR rejections write the reason to `audit_log.reason` via `update_status()`, not to `pr_comments`. The planned `rejection_reason` column on `workflows` (from Topic 7) will fix this. Until then, the pipeline reads from both sources:

- `workflows.pr_comments` for `commented` decisions
- `audit_log` JOIN on `workflow_id` and rejection action for `rejected` decisions

Rejection reasons are particularly useful for the Curator's failure-path audit (Topic 9): they often contain the most explicit statement of what the agent got wrong. "Rejected — you created a new PR instead of updating the existing branch" is a signal that the context store is missing a pattern about branch reuse on re-runs.

## Wiring into the graph

The PR feedback classification step fits as the first sub-step of `run_learning_pipeline`:

```
run_learning_pipeline:
  1. classify_pr_feedback(workflow)       ← parse + type the pr_comments rounds
  2. evaluator(workflow, classified_feedback)
  3. reflector(workflow, classified_feedback)  ← receives pre-typed feedback
  4. curator(candidates)
  5. insert context_extraction_log row
```

The Evaluator benefits from pre-classified feedback: it can short-circuit to `proceed` immediately if any `Correction` or `Omission` items are present, without parsing raw text. The Reflector generates cleaner candidates because the feedback type guides what kind of context item to produce.

---

## What PR feedback alone can't do

PR feedback tells you what the engineer thought of the output — it doesn't tell you whether the agent's *reasoning process* was sound. A plan that happened to produce an accepted PR may have contained subtle reasoning errors that will surface on harder tasks later.

The PR feedback pipeline is a complement to, not a replacement for, the full trace-based learning pipeline. The trace pipeline surfaces reasoning signals (clarification history, planner confidence, outcome divergence from plan). The PR feedback pipeline surfaces evaluation signals. Both are needed for a complete picture.
